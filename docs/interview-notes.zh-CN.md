# nano-job 面试讲解稿

## 一句话版本

`nano-job` 是一个基于 Java 21 和 Spring Boot 实现的小型任务执行平台，支持延迟执行、异步 worker、重试策略、去重、租约恢复、类型化 payload 契约和执行指标。

## 3 分钟版本

我想做一个体量不大、但能体现后端设计能力的项目，所以没有再做 CRUD 管理系统，而是做了一个通用任务执行平台 `nano-job`。

它的主流程是：

1. 客户端提交任务
2. 服务端先校验 payload 契约并执行 dedup 策略
3. 调度器找到到期任务
4. dispatcher 用条件更新领取任务
5. 独立线程池异步执行 handler
6. lifecycle 服务决定任务进入成功、重试等待或最终失败
7. 每次执行都会写 attempt 日志，并记录指标

这个项目的重点不是业务复杂度，而是平台能力：

- payload 契约是注解驱动的
- retry 是按任务类型做策略化设计
- dedup 也是策略化的
- 状态流转集中在 lifecycle 服务
- Web 请求线程和后台执行线程隔离

所以它虽然小，但更像一个真实后端基础设施项目。

## 5 个核心技术点

### 1. 领取幂等

问题：

调度器可能连续多次扫到同一个任务，或者多个线程几乎同时尝试执行它。

做法：

我没有先查后改，而是用数据库条件更新把任务从 `PENDING` 或 `RETRY_WAIT` 原子地切到 `RUNNING`。

价值：

这样同一个任务只能被领取一次，避免重复消费，而且不需要上来就引入复杂分布式锁。

### 2. 租约恢复

问题：

如果 worker 在领取任务后崩掉，任务可能永久卡在 `RUNNING`。

做法：

我给 `RUNNING` 增加了租约拥有者和过期时间，dispatcher 会定期扫描过期租约，并把它重新走失败路径。

价值：

我没有再维护一套独立的恢复模型，而是让租约超时复用现有的重试和终态逻辑，系统语义更统一。

### 3. 按任务类型区分重试策略

问题：

不同任务类型的失败原因不同，不能把“要不要重试”写死在一个通用生命周期引擎里。

做法：

我把重试决策抽成按类型注册的策略。

例如：

- `HTTP` 的 `5xx`、网络失败、超时会重试
- `HTTP` 的 URL 非法、参数非法不会重试
- `NOOP` 使用更简单的线性重试

价值：

状态机保持通用，恢复策略又能跟任务语义绑定。

### 4. Dedup 语义

问题：

提交幂等不只是“相同 key 返回旧任务”这么简单。

做法：

我把 dedup 抽成了策略，并明确了三种结果：

- 创建新任务
- 返回已有任务
- 拒绝创建

同时我还加了：

- 漂移检测：同一个 `dedupKey` 但 `type` 或 `payload` 不同，直接拒绝
- 时间窗口：最近刚结束的任务在一定时间内仍可视为重复提交

价值：

这能同时处理重复点击和语义冲突，而不是简单粗暴地复用旧任务。

### 5. Payload 契约和可观测性

问题：

如果 payload 解析、字段校验和任务类型注册都靠手写，很容易漂移，也不利于扩展。

做法：

我用 Java record + 校验注解定义 payload 契约，再通过注解自动发现任务类型。  
另外我还加了 Micrometer 指标，包括：

- 领取数
- 开始执行数
- 成功数
- 失败数
- 重试数
- 最终失败数
- 租约恢复数
- 当前各状态任务数

价值：

这样平台既是类型安全的，也是可观察、可运维的。

## 面试时我会强调什么

- 我刻意选了一个小型后端基础设施项目，而不是业务 CRUD 项目。
- 这个项目的复杂度主要在任务生命周期控制，而不是界面或表单。
- 我重点处理了重复提交、重复消费、worker 崩溃、外部依赖失败这些真实问题。
- 我在保证项目小而完整的前提下，尽量把规则做成可扩展的策略和契约。

## 如果继续扩展，我会做什么

- 更明确地暴露 actuator metrics
- 加 dashboard 更友好的 tag，比如 HTTP 结果类别、重试原因
- 增加 traceId 或 correlationId
- 增加死信处理
- 调度扫描做分页和批量优化
